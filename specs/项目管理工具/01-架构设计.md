# 01 · 架构设计

## 1. 总体形态

单机本地 Web 应用。一个进程内同时承担:扫描解析、入库、分析、Web 渲染。无前后端分离,无外部服务依赖(SQLite 为本地文件)。

```
┌──────────────────────────────────────────────┐
│                浏览器 (localhost)              │
│   Jinja2 渲染页面 + ECharts 图表(甘特等)      │
└───────────────▲──────────────────────────────┘
                │ HTTP (FastAPI 路由)
┌───────────────┴──────────────────────────────┐
│                  应用进程 (Python)             │
│                                               │
│  web (路由/视图)                               │
│      │ 调用                                    │
│  analytics (延期/撞车/汇总计算)                 │
│      │ 读                                      │
│  storage (SQLite 读写, 快照)  ◄── ingest (扫描+解析)│
│                                   │            │
└───────────────────────────────────┼───────────┘
                                    │ 读文件(只读)
                            ┌───────▼────────┐
                            │ NAS 共享文件夹  │
                            │  *.xlsx 项目表  │
                            └────────────────┘
```

## 4. 数据流(一次"扫描 NAS")

1. web 收到 `POST /scan` → 调用 ingest。
2. `scanner` 列出 NAS 目录下所有 `*.xlsx`,按文件名解析出(业务ID、名称、快照日期);**同一业务ID 同一快照日期保留一份**(若重复,取文件修改时间最新者,并记质检告警)。
3. 对每个待入库文件:`parser` 按 `template_map` 解析 4 个区块 → 产出 `Snapshot`(含 project / tasks / members);`qc` 同步产出该文件的质检问题列表。
4. `storage.repository` 幂等写入:以(业务ID + 快照日期)为快照唯一键,已存在则覆盖更新,不存在则插入;同时写入质检记录。
5. 返回本次扫描汇总(新增/更新/跳过/告警数)给 web,刷新页面。

> 说明:分析(延期/撞车)在**查询时实时计算**,不在入库时固化(避免"今天"变化导致数据过期)。仅"快照原始数据"入库。

## 5. SQLite 表结构

> 设计原则:快照不可变地保留每天的原始解析结果;分析结果不落库(实时算)。所有金额/百分比按原值存,空值存 NULL。

```sql
-- 项目快照主表:一行 = 某业务在某快照日期的基本信息+合同经营
CREATE TABLE snapshot (
    biz_id          TEXT NOT NULL,         -- 业务ID(主键来源, 权威=表内单元格)
    snap_date       TEXT NOT NULL,         -- 快照日期 YYYY-MM-DD(来自文件名)
    project_name    TEXT,                  -- 业务名称
    biz_type        TEXT,                  -- 业务类型
    stage_status    TEXT,                  -- 当前阶段/状态
    customer        TEXT,                  -- 客户名称
    customer_industry TEXT,
    description     TEXT,                  -- 一句话说明
    deliverables    TEXT,                  -- 关键任务/交付内容
    setup_date      TEXT,                  -- 立项日期 YYYY-MM-DD
    plan_deliver_date TEXT,                -- 计划交付日期
    actual_deliver_date TEXT,              -- 实际交付日期(可空)
    progress_pct    REAL,                  -- 完成度%(0-100, 可空)
    risk_level      TEXT,                  -- 风险等级
    risk_desc       TEXT,                  -- 风险描述/难点
    current_issue   TEXT,                  -- 当前主要问题
    -- 合同与经营
    contract_status TEXT,
    contract_no     TEXT,
    contract_amount REAL,
    invoiced_amount REAL,
    received_amount REAL,
    gross_margin_pct REAL,
    owner_name      TEXT,                  -- 项目总负责人
    pm_name         TEXT,                  -- 项目经理
    last_update_date TEXT,                 -- 表内"最后更新日期"
    form_status     TEXT,                  -- 表单状态
    source_filename TEXT,                  -- 来源文件名(留痕)
    ingested_at     TEXT,                  -- 入库时间戳
    PRIMARY KEY (biz_id, snap_date)
);

-- 人员分工:一行 = 某快照里某人一行
CREATE TABLE snapshot_member (
    biz_id      TEXT NOT NULL,
    snap_date   TEXT NOT NULL,
    row_idx     INTEGER NOT NULL,          -- 原表行序, 保证唯一与可追溯
    name        TEXT,
    emp_id      TEXT,
    role        TEXT,
    task_desc   TEXT,
    join_start  TEXT,                       -- 参与开始 YYYY-MM-DD(可空)
    join_end    TEXT,                       -- 参与结束(可空)
    workload_pct REAL,                      -- 投入工作量%(可空)
    progress_note TEXT,
    eval        TEXT,
    note        TEXT,
    is_external INTEGER DEFAULT 0,          -- 是否外协/非真实人员(外协/?/空)
    PRIMARY KEY (biz_id, snap_date, row_idx)
);

-- 阶段/任务:一行 = 某快照里一个阶段或子任务
CREATE TABLE snapshot_task (
    biz_id      TEXT NOT NULL,
    snap_date   TEXT NOT NULL,
    row_idx     INTEGER NOT NULL,
    seq         TEXT,                        -- 原表"#"列(可能重复, 文本存)
    task_name   TEXT,
    owner       TEXT,                        -- 负责人(可能是"外协"或多名字)
    plan_start  TEXT,
    plan_end    TEXT,
    actual_start TEXT,
    progress_pct REAL,
    status      TEXT,                        -- 已完成/进行中/未开始等
    note        TEXT,
    PRIMARY KEY (biz_id, snap_date, row_idx)
);

-- 质检问题:一行 = 一次扫描中针对某文件/某字段的一条告警
CREATE TABLE qc_issue (
    biz_id      TEXT,                        -- 可能解析不出, 允许空
    snap_date   TEXT,
    source_filename TEXT,
    severity    TEXT,                        -- error / warning
    issue_type  TEXT,                        -- 见 02 质检类型枚举
    location    TEXT,                        -- 出错位置(区块/行/字段)
    message     TEXT,                        -- 人类可读说明
    detected_at TEXT,
    PRIMARY KEY (rowid)                       -- 自增
);

-- 扫描批次记录(便于"最近一次扫描结果"展示)
CREATE TABLE scan_run (
    run_id      INTEGER PRIMARY KEY AUTOINCREMENT,
    started_at  TEXT,
    finished_at TEXT,
    files_total INTEGER,
    inserted    INTEGER,
    updated     INTEGER,
    skipped     INTEGER,
    qc_errors   INTEGER,
    qc_warnings INTEGER
);
```

索引建议:
```sql
CREATE INDEX idx_snapshot_latest ON snapshot(biz_id, snap_date);
CREATE INDEX idx_member_person ON snapshot_member(name, snap_date);
CREATE INDEX idx_task_proj ON snapshot_task(biz_id, snap_date);
```

## 6. "最新快照"取数口径

总览/详情/预警等所有视图,默认只看**每个业务的最新快照**:对每个 `biz_id`,取 `MAX(snap_date)` 的那一条及其子表。历史快照只留存,不在第一期展示。

参考 SQL:
```sql
SELECT s.* FROM snapshot s
JOIN (SELECT biz_id, MAX(snap_date) AS m FROM snapshot GROUP BY biz_id) t
  ON s.biz_id = t.biz_id AND s.snap_date = t.m;
```

## 8. 异常与降级原则(全局)

- **单文件解析失败不影响其他文件**:逐文件 try/except,失败文件记 `qc_issue(severity=error)` 并跳过,扫描继续。
- **缺字段不报错**:可空字段缺失存 NULL,页面显示"—"。
- **只读 NAS**:绝不修改/写回源文件。
- **NAS 不可达**:扫描接口返回明确错误提示(路径不可访问),不崩溃。
- 日志:每次扫描输出处理文件数、成功/失败、质检统计。
