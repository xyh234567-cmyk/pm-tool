# 03-3 · 分析模块(analytics)

## 职责
基于 storage 的**最新快照**数据,实时计算:延期、资源撞车、总览/详情/KPI 组装。**分析结果不落库**(依赖"今天"动态变化)。

## 组成
- `delay.py`:延期判定。
- `resource.py`:资源撞车。
- `dashboard.py`:总览行、KPI、详情组装。

## 通用约定
- "今天" = 运行时本地日期(`date.today()`),`YYYY-MM-DD`。
- 日期比较用字符串(同为 `YYYY-MM-DD` 时字典序=时间序)。空日期视为"无",不参与比较。

## A. 延期判定(delay.py)

### 任务级
任务 `t` 延期 ⇔ `t.plan_end` 非空 且 `t.plan_end < today` 且 **未完成**。
- "未完成"定义:`status` 不属于已完成类(不含"已完成/完成") **且** `(progress_pct is None or progress_pct < 100)`。
- 即:已完成的任务即使计划结束已过,不算延期。

```python
def is_task_delayed(t, today) -> bool:
    if not t.plan_end:
        return False
    done = (t.status and t.status.strip() in DONE_STATUSES) or (t.progress_pct == 100)
    return (t.plan_end < today) and not done
```

### 项目级
项目 `p` 延期 ⇔ 满足任一:
1. 存在任一延期任务(任务级判定);**或**
2. `p.plan_deliver_date` 非空 且 `< today` 且 `p.actual_deliver_date` 为空(到期未交付)。

### 距交付天数(overview 用)
`days_to_deliver = (plan_deliver_date - today).days`(plan 空→None)。
- `<0`:逾期(红),展示"逾期N天"。
- `0..deliver_soon_days`:临近(黄)。
- `>deliver_soon_days`:正常。

### 输出
- `get_delayed_tasks() -> list`(项目+任务,带逾期天数)。
- `get_delayed_projects() -> list`(项目+延期原因:任务延期/到期未交付)。

## B. 资源撞车(resource.py)

### 数据源
所有项目**最新快照**的人员行,`is_external=0`,需 `name`、`join_start`、`join_end`、`workload_pct` 齐全。缺任一时间或投入的行 → 不参与精确计算,记 `WORKLOAD_MISSING`(已在解析阶段),并在撞车页"数据不全"区列出。

### 口径(已确认)
同一个人,在**时间重叠**的多个项目里,`workload_pct` 之和 **>100%** → 撞车。单个项目自身 >100% 不算撞车(可作二期"超配"提醒)。

### 算法(扫描线,按人)
```
对每个 person:
  intervals = [(start, end, workload, biz_id, project_name) for 该人各项目行, 字段齐全]
  若 intervals 数 < 2: 跳过(单项目不构成撞车)
  boundaries = 排序去重所有 start 与 (end + 1天)
  for 相邻 boundary 形成的每个时间段 [a, b):
      active = [iv for iv in intervals if iv.start <= a and a <= iv.end]
      total = sum(iv.workload for iv in active)
      if len(active) >= 2 and total > 100:
          记录冲突段: {period:[a, max(active.end)], total, projects:[...]}
  合并该人相邻/重叠的冲突段,取峰值 total
  若有冲突段 → 产出一条 person 级撞车结果:
      { name, peak_total, conflict_periods:[...], projects:[...],
        includes_today: 任一冲突段包含 today }
```
- 区间端点用"日"粒度;`end` 含当天(闭区间),故边界用 `end+1天`。
- `includes_today=True` 的优先级更高(当前正在撞)。

### 输出
- `get_resource_conflicts() -> list[person 级结果]`,按 `includes_today` 优先、`peak_total` 降序。
- `count_conflict_persons() -> int`(KPI 用)。

## C. 组装(dashboard.py)

### KPI(总览顶部 4 卡)
- `在管项目` = 最新快照项目数(排除 `actual_deliver_date` 非空且状态为已结束的?第一期口径:统计所有有最新快照的项目,不剔除已交付;如需剔除已结束,二期再议)。
- `延期` = 延期项目数。
- `资源撞车` = 撞车人数。
- `临近交付` = `0<=days_to_deliver<=deliver_soon_days` 的项目数。

### 总览行
对每个最新快照项目输出:
`{biz_id, project_name, owner_name, stage_status, progress_pct, risk_level, plan_deliver_date, days_to_deliver, delay_flag(bool), deliver_state ∈ {overdue, soon, normal}}`。

### 总览组装
`get_overview() -> {kpi: {...}, rows: [...]}`:内部调用 storage.get_latest_projects、delay、resource,产出上面的 KPI 与总览行。

### 详情
`build_project_detail(biz_id)`(内部先调 `storage.get_project_detail`,再补算延期/颜色/甘特域)返回:
- 基本信息 + 合同经营(主表全字段)。
- `members[]`(含 is_external 标记,撞车命中标记)。
- `tasks[]`(供甘特):每行带 `plan_start/plan_end/actual_start/progress_pct/status` 及 §02-8 的颜色语义 `bar_color ∈ {done, doing, todo}`。
- 甘特时间域:`domain_start = min(plan_start, actual_start)`,`domain_end = max(plan_end)`;today 落点。

## 依赖
- storage(取最新快照);common(dates、enums)。

## 异常/边界
- 空库 → 所有计数为 0,列表为空,页面空态。
- 全部人员缺时间/投入 → 撞车列表空 + "数据不全"提示。
- 日期缺失任务不参与延期(plan_end 空 → 不延期)。

## 验收要点(用样例 + 构造数据)
- 隔离网关:计划交付 06-15、今天 06-21、未交付 → 项目延期=真;"客户验收"未开始且 plan_end 已过 → 任务延期。
- 构造"陈庆阳"在两个时间重叠项目投入 30%+80% → 撞车,peak=110,includes_today 视日期而定。
