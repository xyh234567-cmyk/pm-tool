# 审查清单(设计方审 CodeX 产出用)

> 读者:设计方(审查侧)。与 `AGENTS.md`(CodeX 侧)对称——那份管"怎么写",这份管"怎么审"。
> 用法:每阶段评审对照"通用审查项"逐条过;每抓到一类新坑,append 到"已沉淀教训"。
> 注意:清单是底线,**替代不了"读懂代码、对照意图"**。别变成机械打勾。
> **更重要的心态(别为审而审)**:审查目标是"确信它对",不是凑一张问题清单。**"看过了,没问题,过"是完整合格的结果**,不必非挖出问题才算到位;为证明认真而把芝麻说成西瓜是真失职。力度配风险:核心逻辑/数据层深抠,胶水/样式扫一眼没事就放。

# 第一部分 · 通用审查项(可复用)

## 1. 契约一致性
- 字段映射、枚举、表结构是否**照搬 `contracts/*.yaml`**,有无私自增删改。
- MD 与 YAML 有无冲突;冲突时是否以 YAML 为准。

## 2. 规格符合
- 行为是否符合规格;有没有引入**规格外的字段/行为**。
- 该阶段 `05-启动指令` 的 DoD 是否真的满足。

## 3. 测试覆盖(重点 —— 防 agent 盲点)
- **专门审负例/边界/异常/空值**,不只看正例。
- 警惕"**舒适区盲点**":agent 倾向只测自己好实现的分支,绕开难的/有问题的分支。凡是带条件开关(如 `xxx_only=True/False`)的,两条分支都要有测试。
- 算法类是否**测试先行**;验收例子(规格里给的)是否都落成了测试。

## 4. 自己跑,不靠肉眼
- 实际 `pytest` 跑一遍,别只读测试代码就信它过。
- 必要时构造对抗用例,亲自验算法口径。
- **权威测试 = 用户目标环境(Python 3.12)那次运行**;设计方沙盒(3.10)仅辅助双检,注意版本差异(如 `tomllib` 3.11+ 才有),别拿沙盒结果当数。
- **验证分工**:设计方沙盒只跑逻辑/数据层(纯 Python pytest + 真实样例集成冒烟);**运行态(起服务、浏览器点样式/图表、真实环境集成、git 提交)归用户 3.12 本机**。沙盒装不动重依赖(如 fastapi)是正常的,别硬试——web 的真正验证靠用户点开浏览器。

## 5. 异常与降级
- 有没有**静默吞错/静默丢数据**(该告警的要显式)。
- "只读"约束有没有被破坏(如回写源文件)。
- 单点失败是否降级而非整体崩。

## 6. 反向质疑规格(自我审查)
- bug 可能不在 CodeX,而在**我自己的契约/设计**。审查时对规格本身也保持质疑。
- 发现是规格的洞 → 先改契约/规格,再让 CodeX 改实现,并记 `决策记录.md`。

## 7. 代码卫生
- 死导入、明显冗余、不必要的"重复造轮子"(标准库能干的别自己写)。
- 命名/类型注解/异常说明是否到位(对照 `AGENTS.md`)。

## 8. 口径陷阱(易错清单)
- **符号方向**(如"距今天"vs"距交付"正负相反)。
- 单位、时区、空值的展示(空应显示"—",不是 0/None)。
- 日期字符串比较是否同格式(`YYYY-MM-DD`)。

## 9. 呈现层人眼验(运行态,用户在浏览器扫)
> 呈现层正确性数据测试照不到,只能眼看(见 L8)。每个图表/页面过一遍:
- 顺序/朝向:列表、甘特任务顺序与 mockup 一致(早在上)。
- 关键标记:今天线位置、红黄绿状态色是否对。
- 边界态:空数据显示"—"/空态文案,不是报错或 0/None。
- 格式:日期 YYYY-MM-DD、百分比取整、金额千分位。
- 布局:断网下样式在、无裸奔、无横向溢出。

# 第二部分 · 已沉淀教训(append-only,带阶段来源)

- **L1 · 符号方向(阶段1)**:`days_from_today` 约定"正数=已过去",与"距交付(剩余天数)"方向相反。用到方向相关计算时,先确认正负口径。
- **L2 · schema 留关联(阶段2)**:`qc_issue` 漏了与 `scan_run` 的关联,导致"只看最近一次扫描"只能靠时间戳硬凑、会捞出历史数据(真 bug)。**凡是"属于某批次/某次操作"的明细表,schema 就要留 run_id/批次外键**;审查时反向质疑自己的契约。
- **L3 · 测试舒适区盲点(阶段2)**:CodeX 测了 `latest_run_only=False`、绕开了有问题的 `=True`。**带开关的接口,强制要求两条分支都测**。
- **L4 · 重复造轮子(阶段0)**:CodeX 为"省依赖"手写了 TOML 解析器,而标准库 `tomllib`(3.11+)就有。**审查时检查:是不是该用标准库/成熟库而自己造了轮子**。
- **L5 · 合成数据掩盖解析 bug(阶段3)**:81 个单元测试全绿,但拿真实样例一跑,发现解析器只读 A 列标签、漏掉所有次列字段(plan_deliver/stage_status/risk_level 全 NULL),必填校验也跟着瞎。合成测试数据恰好顺着解析器的错误假设。**铁律:解析/导入类必须有"喂真实样例 + 断言关键字段非空且正确"的集成测试,光看单元测试全绿不算数。**
- **L6 · 验证后再下结论(阶段3)**:审查时一度怀疑文件名正则有 bug,实跑后证明是误报。**对疑似 bug 先用真实数据验证再宣布,别凭脑内推演定罪。**
- **L7 · 未要求的框架/依赖 + 部署环境(阶段5)**:CodeX 自作主张引入 Tailwind Play CDN(没要求、且 mockup 本是纯 CSS),全前端走 CDN——对"本地/离线"工具会页面裸奔、甘特空白。**审查时核对:有没有引入未要求的框架/依赖?依赖是否适配部署环境(离线/内网/隐私)?** 这是 agent 反复出现的倾向(参见 L4 手写 TOML)。
- **L9 · 旧报告/文档会过时(棚改)**:改造已有项目时,既有审查报告、设计文档常与实时代码严重脱节(本例 v3 报告里"131 处 str(e)、双数据源、重复 return、需求 stats 硬编码"等大半已修)。**每个待修项必须先对实时代码复核,再写方案/动手——别按旧报告规划"修早就修好的东西"。**
- **L8 · 呈现层正确性数据测试照不到(阶段5)**:甘特任务顺序画反了(ECharts 分类轴首项在底部,缺 `yAxis.inverse`)。数据顺序是对的、pytest 全过,但**"渲染出来朝哪个方向"数据测试看不见**,只有人眼看渲染才暴露。**呈现层(图表轴向、显示排序、日期/数字格式、配色、截断、空态、布局)必须对照 `mockups/` 用眼验,不能只靠 pytest。** 验收时照 §9 呈现层人眼验清单扫一遍渲染结果。
- **L10 · 特征化/回归测试的陷阱(棚改)**:给已有代码补回归测试时,"锁住当前行为"可能把 bug 当"正确"锁住。本例:R4 给 Excel 解析补测试,`assert contract_amount not in result` 把"合同区把相邻标签当值"这个真 bug 锁成了预期,61 个测试全绿却掩盖缺陷。**断言必须基于"正确行为",不是"当前行为";喂真实样例时,亲手核对每个断言锁的是对的还是现状。审查要审测试本身,不只审被测代码。**
- **L11 · 退役一个概念要 grep 所有层(全栈)**:删/退役一个跨前后端的概念时,别只清你以为它在的那层。本例 ProjectHub 退役 `data_source`:后端清干净了,前端还遗留 3 处消费者(detail.vue 传参、Plan/Members 的 prop+editable 门、task 拖拽门),还把 `'app'` 误写成 `'.app'` 致任务拖不动。**改/删一个概念前,先 grep 前端+后端所有消费者,逐个处理;改完每一层都验,别只验你动的那层。**
- **L12 · 测试注释在自述"测不到真实路径"=规格有矛盾的信号(母版同步器)**:CodeX 实现 scanner 时,规格要求"三件套齐全才算 spoke"又要"缺件报 missing_file"——逻辑互斥致 missing_file 永不可达。CodeX 没按 `AGENTS §1` 停下问,而是写了个绕过真实路径的测试,**测试 docstring 里白纸黑字写"三件套齐全确保不会产生 missing_file"**。审查时:**测试注释/docstring 若在解释"为什么这里测不到真实代码路径""为什么改用别的方式验",几乎一定是规格自相矛盾或有死代码的信号——顺藤回查规格本身(§6),别放过。** 执行方撞见矛盾应停下问,不应绕过测掉。
- **L13 · 入口/编排层零测试,pytest 全绿≠能跑(母版同步器)**:CLI 入口 `main.py` 因一次没删干净的编辑留下重复 `if`,是 `IndentationError`、整个命令跑不起来;但 35 个 pytest 仍全绿,**因为没有一个测试导入/执行 main.py**——单元测试只覆盖了被编排的模块,编排层本身裸奔。**审查时:确认入口/CLI/编排层至少有 import 测试或 dry-run 冒烟测试;全绿只证明被测的那些单元对,不证明程序能启动。改完务必 `py_compile` + 实跑一次入口(§4),别只看 pytest 数字。**
- **L14 · 编排层 apply 路径必须有非 dry-run 测试(母版同步器·二期)**:教训流集成测试(用例 25–28)全是 dry-run,只验"呈现 + 不写";apply 写入路径——含**调用方剥除 `[↑]` 标记的职责**——零覆盖。于是 `run_lesson_up` 漏剥 `[↑]`、把标记原样写进 hub 的真 bug,pytest 仍全绿、复审实跑 apply 才暴露。**L13 管"入口能不能启动",这条管"写入路径写得对不对":凡 dry-run/apply 双模式的编排函数,apply 那条(备份、写文件、以及调用方对数据的预处理如剥标记/规范化)必须有至少一条非 dry-run 测试,断言落盘后的内容正确。dry-run 全绿只证明"没写",不证明"写对"。**
- **L15 · CLI/无 GUI 工具也要先设计操作流(母版同步器·收尾)**:母版同步器需求明确、模块规格齐全、pytest 全绿,但"人怎么操作"从没在设计阶段讨论过——因为方法论的操作流产物(mockup)只为 GUI 准备,CLI 工具掉进缝里,UX 一路漏到代码做完才补 `操作说明`,且紧接着就要往上补一键启动器(机制设计了、操作 UX 没设计的症状)。**铁律:任何工具(含 CLI/无界面)在冻结模块规格前,必须从操作者口吻把"人怎么用它"端到端走一遍——GUI 出 mockup,CLI 出命令/菜单序列;操作流是设计输入,不是事后文档。判据:"需求明确"≠"知道要什么功能",还要"能讲清怎么用它",讲不顺=需求没真明确,回去补。DoD 加一条:照操作流走查一遍能完成目标任务,不只 pytest 全绿。**
- **L16 · pytest 全绿≠本阶段目标落地(母版同步器·GUI 阶段)**:GUI 控制台阶段 CodeX 90/90 全绿,但本阶段核心目标"建项目收成单一实现 + `.command` 改 shim"根本没落地——`.command` 没改、`python -m app.bootstrap` 入口实跑就报错、连那条"shim 集成测试"都是 inline 绕过真实入口的假绿(L12 类)。绿灯只证明"被测的单元对",不证明"这阶段要达成的事做到了"。**铁律:审查收尾必须回看"本阶段规格的目的/DoD 有没有逐条真落地",拿本阶段意图对照实物点验,别被全绿数字收买。与 L13(入口能不能启动)、L14(写入路径对不对)同源——全绿是必要不充分;凡"改了 A 才算达成"的目标,必须有一条直接验 A 的检查(实跑入口 / 实物 grep / 端到端走查),不能只靠旁证的单元测试。**
