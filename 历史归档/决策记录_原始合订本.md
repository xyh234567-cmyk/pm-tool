## 2026-06-21 · 协作方式

- **分层规格 + 不写代码**:设计方只产出分层规格文档,代码由 CodeX 生成。理由:职责分离,设计方专注顶层。
- **格式分工**:MD 讲思路、YAML 锁硬契约、Mermaid 画结构、pytest 锁正确性、Git 管历史。理由:各取所长,机器须照搬的部分用确定性格式,降低理解偏差。
- **HTML mockup**:UI 拍板后导出独立 HTML mockup 存 `specs/<项目>/mockups/`,兼作 Jinja2 模板雏形。理由:直观、可留存、降低 UI 走样。
- **质疑力度调高 + 批量提问 + 关键节点复述**(关键节点定义见 CLAUDE.md 行为准则 6)。

## 2026-06-21 · 项目管理工具 · 第一期范围

- **第一期 = 跨项目全局监控看板**,5 视图:总览、详情(含甘特)、延期预警、资源撞车、数据质检。理由:用户选定的最痛点;最轻、见效最快。
- **明确不做(进路线图)**:趋势视图、全生命周期串联、需求管理、售后管理、文档归档/知识沉淀、多人协同/权限、系统内编辑回写。理由:防膨胀,守住"精简"初心。
- **协作模式 A**:他人只交 Excel,仅 PMO 一人进系统。理由:单人监控,无需账号/权限,复杂度低一档。

## 2026-06-21 · 技术形态

- **本地 Web + 服务端渲染**(FastAPI + Jinja2 + ECharts)+ 本地 SQLite。理由:跨平台、对基础开发者最易维护自调。
- **采集触发 = 手动扫描**。理由:最简单可控;定时自动放二期。
- **保留每日快照历史**(按 业务ID+快照日期)。理由:用户要看趋势;快照天然支持,二期直接取数。

## 2026-06-21 · 数据口径(会被反复引用,重点)

- **主键 = Excel 表内"业务ID"单元格**(标准格式 `RW{年}-{3位}`,如 RW2026-001),权威源;文件名仅供人阅读。
- **快照日期 = 文件名【YYYYMMDD】**;无法解析则回退文件 mtime 并告警。
- **表内ID 与文件名ID 不一致**:以表内为准,并进数据质检报告。
- **延期口径**:任务 plan_end < 今天且未完成 → 任务延期;项目含延期任务或(计划交付已过且未实际交付)→ 项目延期。
- **资源撞车口径**:同一人、时间重叠的多个项目(≥2)投入工作量%之和 >100% → 撞车;单项目自身>100% 不算撞车(二期再议"超配")。
- **解析策略**:靠"区块锚点 + 标签匹配"定位,不写死行号列号;脏数据不静默丢弃,一律进质检报告。

## 2026-06-21 · 架构边界:监控模型 vs 业务系统模型(关键,防误接)

- **第一期是"监控模型"**:核心假设=「一份 Excel=一个项目的每日快照,只读、不回写」,所有表按 `(业务ID+快照日期)` 存快照。它为"跨项目监控"而造,**不是业务系统(system of record)**。
- **扩展不能 bolt-on,要走"实体核重构"**:需求管理/售后/全生命周期串联这类模块有自己的生命周期、要在系统内录入编辑、要跨阶段挂链——当前**无写路径、无一等实体**,塞不进快照表。真做那天,应引入"项目/人/需求/工单皆一等实体"的关系核,让 Excel 快照采集退化为其一个数据源、监控只是一个视图。这是第二期的数据层架构决策,不是加模块。
- **资源管理半支持**:数据(人员投入%/起止)已在采,但"人"是按姓名字符串匹配、非一等实体。**便宜的早期动作:从现在起把 Excel"员工ID"列填起来**(现在常空),稳定人员身份,不需改任何架构。
- 取舍说明:第一期不为这些预留架构=刻意的 YAGNI/防膨胀,非缺陷;但须睁眼进,别误以为"监控模型=业务系统模型"往错方向接。
