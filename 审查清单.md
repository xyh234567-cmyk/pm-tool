# 第一部分 · 通用审查项(可复用)

## 1. 契约一致性
- 字段映射、枚举、表结构是否**照搬 `contracts/*.yaml`**,有无私自增删改。
- MD 与 YAML 有无冲突;冲突时是否以 YAML 为准。

## 2. 规格符合
- 行为是否符合规格;有没有引入**规格外的字段/行为**。
- 该阶段 `05-启动指令` 的 DoD 是否真的满足。

## 3. 测试覆盖(重点 —— 防 agent 盲点)
- **专门审负例/边界/异常/空值**,不只看正例。
- 警惕"**舒适区盲点**":agent 倾向只测自己好实现的分支,绕开难的/有问题的分支。凡是带条件开关(如 `xxx_only=True/False`)的,两条分支都要有测试。
- 算法类是否**测试先行**;验收例子(规格里给的)是否都落成了测试。

## 4. 自己跑,不靠肉眼
- 实际 `pytest` 跑一遍,别只读测试代码就信它过。
- 必要时构造对抗用例,亲自验算法口径。
- **权威测试 = 用户目标环境(Python 3.12)那次运行**;设计方沙盒(3.10)仅辅助双检,注意版本差异(如 `tomllib` 3.11+ 才有),别拿沙盒结果当数。

## 5. 异常与降级
- 有没有**静默吞错/静默丢数据**(该告警的要显式)。
- "只读"约束有没有被破坏(如回写源文件)。
- 单点失败是否降级而非整体崩。

## 6. 反向质疑规格(自我审查)
- bug 可能不在 CodeX,而在**我自己的契约/设计**。审查时对规格本身也保持质疑。
- 发现是规格的洞 → 先改契约/规格,再让 CodeX 改实现,并记 `决策记录.md`。

## 7. 代码卫生
- 死导入、明显冗余、不必要的"重复造轮子"(标准库能干的别自己写)。
- 命名/类型注解/异常说明是否到位(对照 `AGENTS.md`)。

## 8. 口径陷阱(易错清单)
- **符号方向**(如"距今天"vs"距交付"正负相反)。
- 单位、时区、空值的展示(空应显示"—",不是 0/None)。
- 日期字符串比较是否同格式(`YYYY-MM-DD`)。

# 第二部分 · 已沉淀教训(append-only,带阶段来源)

- **L1 · 符号方向(阶段1)**:`days_from_today` 约定"正数=已过去",与"距交付(剩余天数)"方向相反。用到方向相关计算时,先确认正负口径。
- **L2 · schema 留关联(阶段2)**:`qc_issue` 漏了与 `scan_run` 的关联,导致"只看最近一次扫描"只能靠时间戳硬凑、会捞出历史数据(真 bug)。**凡是"属于某批次/某次操作"的明细表,schema 就要留 run_id/批次外键**;审查时反向质疑自己的契约。
- **L3 · 测试舒适区盲点(阶段2)**:CodeX 测了 `latest_run_only=False`、绕开了有问题的 `=True`。**带开关的接口,强制要求两条分支都测**。
- **L4 · 重复造轮子(阶段0)**:CodeX 为"省依赖"手写了 TOML 解析器,而标准库 `tomllib`(3.11+)就有。**审查时检查:是不是该用标准库/成熟库而自己造了轮子**。
