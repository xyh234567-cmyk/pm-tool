# 决策记录(append-only)

> 规则:只追加,不删改历史。每条记录:决定了什么 / 为什么 / 日期。
> 这是口径的唯一权威出处,避免散落在对话里被遗忘或重复争论。

## 2026-06-21 · 协作方式

- **分层规格 + 不写代码**:设计方只产出分层规格文档,代码由 CodeX 生成。理由:职责分离,设计方专注顶层。
- **格式分工**:MD 讲思路、YAML 锁硬契约、Mermaid 画结构、pytest 锁正确性、Git 管历史。理由:各取所长,机器须照搬的部分用确定性格式,降低理解偏差。
- **HTML mockup**:UI 拍板后导出独立 HTML mockup 存 `specs/<项目>/mockups/`,兼作 Jinja2 模板雏形。理由:直观、可留存、降低 UI 走样。
- **质疑力度调高 + 批量提问 + 关键节点复述**(关键节点定义见 CLAUDE.md 行为准则 6)。

## 2026-06-21 · 项目管理工具 · 第一期范围

- **第一期 = 跨项目全局监控看板**,5 视图:总览、详情(含甘特)、延期预警、资源撞车、数据质检。理由:用户选定的最痛点;最轻、见效最快。
- **明确不做(进路线图)**:趋势视图、全生命周期串联、需求管理、售后管理、文档归档/知识沉淀、多人协同/权限、系统内编辑回写。理由:防膨胀,守住"精简"初心。
- **协作模式 A**:他人只交 Excel,仅 PMO 一人进系统。理由:单人监控,无需账号/权限,复杂度低一档。

## 2026-06-21 · 技术形态

- **本地 Web + 服务端渲染**(FastAPI + Jinja2 + ECharts)+ 本地 SQLite。理由:跨平台、对基础开发者最易维护自调。
- **采集触发 = 手动扫描**。理由:最简单可控;定时自动放二期。
- **保留每日快照历史**(按 业务ID+快照日期)。理由:用户要看趋势;快照天然支持,二期直接取数。

## 2026-06-21 · 数据口径(会被反复引用,重点)

- **主键 = Excel 表内"业务ID"单元格**(标准格式 `RW{年}-{3位}`,如 RW2026-001),权威源;文件名仅供人阅读。
- **快照日期 = 文件名【YYYYMMDD】**;无法解析则回退文件 mtime 并告警。
- **表内ID 与文件名ID 不一致**:以表内为准,并进数据质检报告。
- **延期口径**:任务 plan_end < 今天且未完成 → 任务延期;项目含延期任务或(计划交付已过且未实际交付)→ 项目延期。
- **资源撞车口径**:同一人、时间重叠的多个项目(≥2)投入工作量%之和 >100% → 撞车;单项目自身>100% 不算撞车(二期再议"超配")。
- **解析策略**:靠"区块锚点 + 标签匹配"定位,不写死行号列号;脏数据不静默丢弃,一律进质检报告。

## 2026-06-21 · 代码执行环境与文件约定

- **代码生成 agent 锁定 CodeX**(底层模型 DeepSeek)。理由:已有现成环境;CodeX 不自动加载 `CLAUDE.md`,与"设计契约 / 代码规则"两部宪法分离的设计天然契合,无命名冲突。
- **两部宪法分工**:`CLAUDE.md` = 设计阶段(用户↔设计方)协作契约,读者是设计方;`AGENTS.md` = 执行阶段 CodeX 代码生成/修改规则,读者是 CodeX。两者同 repo 共存,CodeX 只读 `AGENTS.md`。
- **全文本化交付**(MD/YAML/Mermaid/HTML mockup)保留为原则:不依赖"读图"能力,任意代码 agent 皆可消费。

## 2026-06-21 · 仓库结构(阶段0 审查确认)

- **每个工具一个顶层目录**:代码在 `设计/<工具名>/app/...`,其规格在 `设计/specs/<工具名>/`。
- **测试与代码同级**:`设计/<工具名>/tests/`,样例素材放 `tests/fixtures/`(`.gitignore` 对该路径放行 xlsx)。理由:阶段0 中 fixtures 一度错放在仓库根 `tests/`,与代码不同级导致测试找不到素材,已对齐。

## 2026-06-21 · 提交时机

- **commit = 已评审通过的检查点**:CodeX 不自行 commit/push,提交由人工在每阶段评审通过后进行。理由:Git 的价值在"可回滚到已知good点 + 看口径变更历史";自动提交未评审代码会让 history 变流水账、失去"已知good"保证。粒度按阶段,不按每笔编辑。已写入 `AGENTS.md §6`。

## 2026-06-21 · qc_issue 关联扫描批次(阶段2 评审修正)

- **`qc_issue` 增加 `run_id` 字段**,关联 `scan_run.run_id`。理由:原 schema 漏了质检问题与扫描批次的关联,导致"只看最近一次扫描的质检"只能靠 detected_at 时间戳硬凑、会把历史问题一起捞出(真 bug)。
- **扫描批次改为"开始即建、结束回填"**:`start_scan_run()` 先拿 run_id → 期间 `insert_qc(issues, run_id)` → `finalize_scan_run(run_id, stats)`。`get_qc_issues(latest_run_only=True)` 按 `run_id = MAX(run_id)` 精确过滤。已更新 `db_schema.yaml`、`03-1`、`03-2`。
