# 第一部分 · 通用代码规则(可复用)

## 1. 消费规格
- `specs/` 是**唯一事实源**。代码必须以规格为准,不以对话记忆或猜测为准。
- 按规格指定的读取顺序消费;`contracts/*.yaml` 是**不可改的硬契约**(字段映射、枚举、表结构、配置),一个字段都不许擅自增删改。
- **规格有歧义、缺失或自相矛盾 → 立即停下、列出疑问,不许猜着写。** 宁可问,不要脑补。
- `mockups/` 里的数据和脚本**仅供视觉参考**,不是需求;逻辑/字段以规格为准。

## 2. 编码约定
- 主语言 Python;按架构文档的目录结构落地,不自创结构。
- **可读、可自调优先**:面向基础开发者维护;不为优雅过度抽象,也不刻意压低复杂度,按需定复杂度。
- **YAGNI**:不加没要求的功能/灵活性/可配置项,不为不会发生的情况提前设计;明显多于所需就砍到够用。
- 函数/变量命名见名知意;公共函数写类型注解和简短 docstring(说明输入输出与异常)。
- 不引入规格外的第三方依赖;确需引入先在 PR/说明里讲清理由。

## 3. 测试优先(pytest)
- **算法类需求先写测试、再写实现**:把规格里的验收例子先落成 pytest 用例,再实现到测试通过。
- **必须覆盖规格/阶段指令里列明的负例、边界、空值、脏数据**,不许只测好实现的正例。
- **带开关的接口(如 `xxx_only=True/False`)两条分支都要有测试**,不许绕开难的那条。
- 正确性靠跑测试确认,不靠肉眼。

## 4. 完成定义(DoD)
一项工作算完成,需同时满足:
- 相关 pytest 全通过;
- 行为与规格一致,未引入规格外的字段/行为;
- 异常路径按规格处理(不静默吞错、不静默丢数据);
- 代码可运行、结构清晰。

## 5. 红线(通用)
- 不擅自扩展需求范围;冒出的新点子写进 `路线图.md`,交回设计方,不当场塞进当前实现。
- 不静默吞异常;该告警/记录的必须显式处理。
- 规格标"只读"的数据源,绝不写回或修改。
- 不删改 `决策记录.md` 的历史(只追加)。

## 6. 提交与闭环
- **不自行 commit / push。** 提交由人工在"评审通过"后进行;一次 commit = 一个已评审通过的检查点。CodeX 只管把改动写到工作区即可。
- 代码与 `specs/` 同一 repo,保持同步。
- 生成/修改后,把结果(代码结构或关键片段、报错)交回设计方,对照规格审查再迭代——**闭环不能断**。

## 7. 外科手术式改动(改既有代码时)
- 只改为满足当前需求/评审意见必须改的地方;每处改动都要能追溯到一条具体要求。
- 不顺手重构、不重排格式、不翻新没坏的旧代码;跟随既有风格,哪怕你偏好别的写法。
- 不删除没让你动的既有内容;看到无关冗余,提一句、别擅自删。
- 改 bug 时:先写一个能复现该 bug 的测试,再让它通过。

# 第二部分 · 项目专属(项目管理工具 · 第一期)

> 换项目时,只替换本部分,指向新项目的 specs 目录。

## 规格位置与读取顺序
- 规格根:`specs/项目管理工具/`
- 顺序:`00-总览` → `01-架构设计` → `02-数据模型与Excel解析规格` → `03-模块规格/`(1采集→2存储→3分析→4Web)→ `04-页面与交互规格`。

## 固定技术栈(不得替换)
FastAPI + Jinja2(服务端渲染)+ ECharts + SQLite(sqlite3)+ openpyxl;uvicorn 启动。

## 项目专属红线
- **主键 = Excel 表内"业务ID"单元格**(`RW{年}-{3位}`);文件名仅供人读与取快照日期。
- 源 Excel **只读**,绝不回写。
- 解析靠"**区块锚点 + 标签匹配**"定位,**不写死行号列号**;脏数据全部进**质检报告**,不静默丢弃/算错。
- 分析结果(延期/撞车/距交付)**实时计算、不落库**;只把每日快照原始数据入库。
- 延期、资源撞车口径以 `03-3-分析模块.md` 与 `决策记录.md` 为准,不得自行改口径。

## 项目专属 DoD
- 用样例 Excel 能跑通"扫描→解析→入库→5视图展示"。
- `03-3` 里的验收例子(隔离网关判延期、构造撞车场景)有对应 pytest 且通过。
- 旧格式样例(业务ID=125)能入库并产出对应质检告警。
