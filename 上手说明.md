# 上手说明(从零起一个新项目)

> 给未来的你 / 同事:照这份就能用这套"顶层设计 + AI 编码"的流程起一个新工具。
> 想懂"为什么这么干",看 `方法论精要.md`;想要完整规则,看 `CLAUDE.md` / `AGENTS.md` / `审查清单.md`。这份只讲"具体怎么做、按什么顺序"。

## 这是什么 / 什么时候用
你要做一个自用小工具,但**你只做顶层设计(需求、UI、架构、模块规格),代码交给 AI(CodeX)生成**。适合"想法清楚、但不想自己一行行写代码"的场景。

## 三个角色
- **你**:拍板需求、跑测试、在浏览器验、做 git 提交。
- **设计方 AI(Claude)**:质疑需求、出分层规格、审 CodeX 的代码、给修复清单。**不写代码。**
- **CodeX**:按规格生成/改代码。**不自行 commit。**

## 第 0 步 · 起项目(一次性)
1. 新建项目文件夹。
2. 从上一个项目把**通用层**拷过来:`方法论精要.md`、`CLAUDE.md`、`AGENTS.md`、`审查清单.md`(这四份基本不动),以及空的 `决策记录.md`、`路线图.md`。
3. `git init`,把上面这些先提一个初始 commit。
4. 跟设计方 AI 说"开始",它会先读这些文件,再开始澄清你的需求。

## 怎么跑(主循环)
1. **需求澄清**:你抛想法,AI 反问到底(场景、边界、什么不做)。新念头一律先进 `路线图.md`,不当场塞进当前期。
2. **方案论证**:AI 给 1–2 个方向 + 取舍,你拍板。
3. **架构 + UI**:数据模型、模块划分用文字/Mermaid 定;UI 用内嵌预览讨论,**拍板后导出 HTML mockup** 存 `specs/<项目>/mockups/`(它既是视觉参考,也是页面的"标准答案")。
4. **出规格**:AI 写分层规格到 `specs/<项目>/`——`00总览/01架构/02数据/03模块/04页面/05启动指令`;机器须照搬的硬契约抽成 `contracts/*.yaml`。
5. **分层生成 → 审 → 修**(核心,见下)。
6. **沉淀**:每阶段评审收尾,AI 判定有无可沉淀教训,有就进 `审查清单`,无就明说。

## 第 5 步细节 · 一层一审(别批量!)
按层分组生成:**数据层(storage/ingest)→ 分析 → 页面**。每组:
1. 你对 CodeX 说:"按 `specs/<项目>/05-启动指令` 做阶段 N,完成即停。"(算法层提醒它测试先行)
2. CodeX 生成完(代码写进项目文件夹)。
3. **你本机**:跑 `pytest`;运行态的(起服务、浏览器点)自己看。
4. 跟设计方 AI 说"阶段 N 好了"。它**直接读你文件夹里的代码**对照规格审,给你:哪里对、哪里偏、要不要改。
5. 过了 → 你 `git commit`(一次 commit = 一个评审通过的检查点)。没过 → 把 AI 的修复意见原样转给 CodeX 改,回到第 3 步。

## 验证分工(谁验什么)
- **设计方 AI 沙盒**:只跑逻辑/数据层(纯 Python pytest + 真实样例集成冒烟)。装不动 fastapi 这类重依赖是正常的。
- **你本机(目标环境,如 Python 3.12)**:权威测试在这跑;起服务、浏览器看样式/图表/空态、git 提交,都在这。
- **呈现层(图表轴向、排序、格式、配色、空态)只能人眼对着 mockup 看**——照 `审查清单 §9` 那张小清单扫。

## 几条铁律(踩过坑换来的)
- **不批量**:一层一审,别攒一大坨(批量会让 bug 牵连面变大)。
- **审是为确信,不是挑刺**:"看过没问题,过"是合格结果,别硬挖芝麻。
- **不为优雅过度设计**:按需定复杂度;新功能进路线图,别现场加。
- **吃数据形状的规格,交手前拿真实样例干跑**:解析/字段映射这类,先程序化验一遍再给 CodeX。
- **离线优先**:本地工具不依赖外网 CDN,前端资源本地化。
- **扩展别 bolt-on**:监控类工具想长成业务系统(需求/售后/全生命周期),要走"实体核重构",不是往现有表上接(见架构边界)。

## 常见运维坑
- 改了 `config.toml` **必须重启服务**(配置启动时读一次缓存住)。
- 端口被占(`Address already in use`):`lsof -ti:<端口> | xargs kill -9` 再起。
- 关服务用 `Ctrl+C`,别直接关窗口(进程可能赖在后台占端口)。
